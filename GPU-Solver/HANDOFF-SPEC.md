## 0. 목표
- GPU 챌린지를 **자동으로 통과**시키고, 통과본을 **런타임 속도 상위권**까지 끌어올린다.
- 신규 문제·(추후) 사내 과제로 일반화.

## 1. 시스템 형태
- **모델 비종속 오케스트레이터** (Hermes 하네스 + 교체 가능한 LLM 백엔드 어댑터).
- 구현 형태 = **오케스트레이터 + (추후) RAG**, 파인튜닝 없음.
  - MVP: RAG 없이 LLM + 검증 루프만.
  - 2차: 코퍼스 쌓이면 RAG 도입(통과 답안·패턴 노트 검색).
  - 파인튜닝 = 장기 옵션(로컬 모델 이전 시점, 코퍼스 충분할 때만).
- 플러그인/MCP = 별개 종이 아니라 "기존 에이전트 확장" 방식 → 고도화 단계에서 선택적 부착.
- 측정·채점은 **LeetGPU 원격 환경**에 의존. 로컬 GPU 불필요.

## 2. LLM 폴백 사다리 (순차, 속도/정확성 우선)
- **코딩 루프 = API 모델만** (단일 API로 깨끗하게). 구독+Claude Code 경유는 루프에 넣지 않음
  (이유: Hermes가 매 쿼리 컨텍스트 전부 주입 / 결과 재처리 토큰 낭비 / 세션 전환 시 수정이력 망각).
- Tier 1: DeepSeek 또는 Codex 중 1개 (초반 테스트로 결정)
- Tier 2: Tier 1에서 안 쓴 나머지
- (Tier 3) Claude는 **코딩 루프에서 제외**. 필요 시에만 *직접 API*로 한정 합류(드물어 비용 제한적) — 선택.
- 승급 트리거: 정확성 반복 실패 / 컴파일 반복 실패
- 각 시도 결과 로깅: 문제ID, Tier, 통과여부, 시도횟수, 실패유형

## 2.5 Claude 역할 (코딩 외)
- 구독(채팅 + Claude Code)으로 **비코딩 작업** 전담: 기획·방향 브레인스토밍, 문서 관리/큐레이션,
  수집 데이터 정규화·문제 매핑 검수, 디버깅 원인 분석 토론, 코드 리뷰·설명(생성 아님).
- → Hermes 자동 루프 마찰(컨텍스트 재주입·세션 망각)을 원천 회피.

## 3. 검증 게이트 (정확성)
- 방식: **출력값 비교** (코드 비교 아님).
  1. GPU 커널 실행 → 출력 배열
  2. 독립 numpy/CPU 레퍼런스 → 기대 출력
  3. 부동소수점 오차 허용범위 내 비교
- 레퍼런스 출처:
  - MVP(A): AlphaGPU `challenge.py`의 레퍼런스+테스트 사용 (학습/연구 범위)
  - 사내 필요 시(C): 해당 문제만 **자체 numpy 레퍼런스 + 테스트 생성기** 제작
    (challenge.py 비복제, 문제 개념에서 독립 구현 → ND 회피)
  - 자체 레퍼런스는 생성 후 challenge.py 결과와 1회 교차검증해 신뢰 확보

## 4. 속도 최적화 (2단 게이트의 2단계)
- 통과한 답안만 진입.
- **정책(토큰 절약)**: 한 방에 통과한 문제 → 즉시 속도 토너먼트 / 고생해서 통과 → 보류 큐.
- 병목 지목 = LeetGPU Pro percentile + Perfetto 트레이스 → 다음 프롬프트에 주입.
- 회귀 방지: 이전 최속 기록보다 느려지면 롤백(챔피언/도전자).

## 5. 독창성
- **near-duplicate만 차단** (토큰/AST 유사도 트립와이어). 완전 구별 강제 안 함.
- 코드 직접 참조보다 *추상 패턴* 참조 우선.

## 6. 수집 / 정규화
- 답안 소스: 개인 GitHub 레포(1차), LeetGPU 사이트 제출(2차, 크롤링 보류).
- 대표 레포 3~4개만 열어 공통 필드 추출 → **필드 포화 시점에 스키마 확정**.
- 레코드 키 = **(문제 × 칩/GPU 타입 × 프레임워크)**.
- 필드(잠정): 문제ID / 출처 / 코드 / 프레임워크 / 칩타입 / 라이선스 / 검증상태 / 런타임 / 기법태그.
- `no_solution` = 정식 상태.
- 문제 매핑 = 유사도 자동 매핑 → 사람 검수.
- 수집 코드는 자체 검증 게이트 재통과해야 신뢰.

## 7. MVP 범위 / 성공 기준
- 범위: **Easy 문제 + 개인레포 답안 + 검증 루프 + 최소 하네스/루프**.
- 성공: **목표 90% / 합격선 70~80% 자동 통과 + 실패 로그 수집**.
- MVP 제외(2차로): 사이트 크롤링, 속도 최적화, 원리 노트, Pro, 멀티에이전트/MCP/스킬/고급 루프/퓨전.

## 8. 라이선스 (주의)
- AlphaGPU = **CC BY-NC-ND 4.0** → 상업/재배포/파생 금지. 사내 활용은 우회안(C) 또는 별도 허가.
- 개인 레포는 라이선스 메타 기록 필수.
- (법률 자문 아님)

## 9. 보조 도구 / 지식 노트
- Godbolt = 컴파일/PTX·SASS 디버깅 보조(필수 아님).
- Discussion(문제 오류) = 특정 문제 안 풀릴 때만 수동 참고.
- **패턴 노트 = solution에서 추출·추상화** (예: "이 문제군은 shared memory 타일링 사용"). RAG 1차 검색 대상.
- **원리 노트 = 이론 기반** (예: "코얼레싱이 왜 빠른가"). 속도 단계에서 도입. 출처가 패턴 노트와 다름(별개).

## 9.5 저장소 구조 (권장)
- `repo/src/` — 하네스·오케스트레이터 코드
- `repo/docs/` — 사양·스키마 (이 문서들). 코드와 함께 git 버전관리(단일 진실 출처).
- `data/` — 수집 코퍼스·로그 (데이터, 코드 아님). 별도 폴더/레포 또는 gitignore/LFS.
- 시작은 한 레포 단순 구성 → 데이터 무거워지면 `data/` 분리.

### 수집물 저장 위치 (중요)
- **원시 수집물(git clone 레포·스크랩 HTML·원본 코드) → WSL2 네이티브 워크스페이스** (`~/gpu-solver/data/` 등). 절대 Obsidian 볼트에 넣지 말 것.
  - 이유: ① 볼트 인덱싱 폭주(앱 저하) ② clone 레포 .git 중첩 충돌 ③ `/mnt/c` 접근이 WSL2에서 다파일 IO에 느림.
- **정제 산출물(정규화 레코드·패턴 노트 등 사람이 읽을 것)만 → Obsidian 볼트.**
- 원칙: **기계용 벌크 = 워크스페이스 / 사람용 노트 = 볼트** (이중 저장의 물리적 구현).

## 9.6 동시 편집 / 충돌 규칙
- **소유권 분리(공간)**: `docs/` = 기획(Cowork/Obsidian) 소유, `src/`+`data/` = Hermes 소유. 남의 영역은 읽기만.
- **git 규율**: 편집 전 `pull`, 편집 후 즉시 `commit & push`. 작은 단위로 자주 커밋.
- **브랜치**: 긴 병렬 작업은 브랜치를 따고 머지로 의식적 통합.
- **한 번에 한 곳**: 같은 파일을 두 앱에서 동시 편집 금지. 전환 시 한쪽 저장·닫고 이동.
- (선택) frontmatter `status` 플래그로 "작업 중" 소셜 락 표시.
- 규모상 **공간 분리 + 편집전 pull/후 commit** 두 개면 충분. 복잡한 락 불필요.

## 10. 착수 순서 (체크리스트)
1. [ ] 대표 개인 레포 3~4개 조사 → 공통 필드 확정
2. [ ] 정규화 스키마 + 저장소 구조 확정
3. [ ] 검증 게이트 구현 (challenge.py 기반)
4. [ ] 최소 하네스 + 최소 루프(생성→제출→pass) 구현
5. [ ] Tier 1 모델 선정용 사전 테스트
6. [ ] Easy 문제 자동 통과 루프 가동 → 성공률·실패로그 측정
